# Work Package: M6 - 신규 회원 & 출시 준비

**Milestone:** M6
**목표:** 신규 회원 유입 경로를 확보하고 안정화하여 정식 출시한다
**기간:** 2주
**선행 조건:** M4, M5 완료

---

## 1. Work Package 개요

```
Phase 1: 후킹 랜딩페이지
    ↓
Phase 2: 신규 회원 플로우
    ↓
Phase 3: 자격 체크 & 유도
    ↓
Phase 4: 안정화 & 출시
```

**수직 분할 원칙:** 각 Phase 완료 시 신규 회원 유치 또는 서비스 안정성 관련 기능이 추가됨

---

## 2. Phase 상세

### Phase 1: 후킹 랜딩페이지

**목표:** 신규 방문자를 끌어들이는 매력적인 랜딩페이지를 만든다
**예상 소요:** 4~5일

#### 작업 목록

| # | 작업 | 산출물 | 완료 기준 |
|---|------|--------|----------|
| 1.1 | 랜딩페이지 레이아웃 | `app/landing/page.tsx` | 기본 구조 |
| 1.2 | 브랜드 스토리 섹션 | 랜딩 UI | 지독해 소개 |
| 1.3 | 지독해 분위기 소개 섹션 | 랜딩 UI | 모임 분위기 전달 |
| 1.4 | 후기 표시 섹션 | 랜딩 UI | 공개 동의 후기만 |
| 1.5 | 공개 동의 후기 조회 API | API | is_public=true 필터 |
| 1.6 | CTA 버튼 ("시작하기") | 랜딩 UI | 회원가입 유도 |
| 1.7 | SSR 적용 | Next.js | 서버 사이드 렌더링 |
| 1.8 | SEO 메타태그 설정 | `metadata` | title, description, og |
| 1.9 | 반응형 디자인 | 랜딩 UI | 모바일 최적화 |
| 1.10 | 스크롤 애니메이션 | Framer Motion | 섹션별 등장 효과 |

#### Scenario (검증 기준)

**SC-M6-001: 랜딩페이지 접근** [Critical]
- Given: 비로그인 사용자
- When: /landing 또는 / 접근 (비로그인 시)
- Then: 브랜드 스토리, 분위기 소개, 후기가 표시됨

**SC-M6-002: 공개 후기 표시** [Critical]
- Given: 공개 동의한 후기 존재
- When: 랜딩페이지 후기 섹션 조회
- Then: 공개 동의한 후기만 표시

**SC-M6-003: SEO 최적화** [High]
- Given: 검색 엔진 크롤러 접근
- When: 페이지 소스 확인
- Then: 메타태그, OG 태그, 콘텐츠가 SSR로 렌더링됨

**SC-M6-004: CTA 클릭** [Critical]
- Given: 랜딩페이지에서 "시작하기" 버튼
- When: 클릭
- Then: 회원가입 페이지로 이동

**SC-M6-005: 모바일 반응형** [High]
- Given: 모바일 기기 (320px)
- When: 랜딩페이지 접근
- Then: 모든 섹션이 정상적으로 표시됨

#### 사용자 가치
> "신규 방문자가 지독해의 매력을 느끼고 가입하고 싶어진다"

### Phase 2: 신규 회원 플로우

**목표:** 신규 회원이 첫 모임까지 자연스럽게 안내된다
**예상 소요:** 3~4일

#### 작업 목록

| # | 작업 | 산출물 | 완료 기준 |
|---|------|--------|----------|
| 2.1 | is_new_member 플래그 | users 테이블 | 신규 회원 구분 |
| 2.2 | 모임 클릭 시 안내 팝업 | 모임 상세 | 신규 회원용 |
| 2.3 | [후킹페이지 보러가기] 버튼 | 팝업 UI | 랜딩페이지 이동 |
| 2.4 | [바로 신청하기] 버튼 | 팝업 UI | 신청 플로우 진입 |
| 2.5 | 첫 모임 전날 환영 알림 | 알림 함수 | D-1 환영 메시지 |
| 2.6 | 첫 모임 후 후기 요청 알림 | 알림 함수 | 종료 후 알림 |
| 2.7 | 다음 모임 안내 알림 | 알림 함수 | 추천 모임 안내 |
| 2.8 | 첫 정기모임 참여 완료 시 전환 | 비즈니스 로직 | is_new_member=false |
| 2.9 | 신규 회원 온보딩 가이드 | UI (선택적) | 서비스 안내 |

#### Scenario (검증 기준)

**SC-M6-006: 신규 회원 팝업** [Critical]
- Given: 신규 회원(is_new_member=true)이 로그인
- When: 모임 상세 페이지에서 신청 버튼 클릭
- Then: "지독해가 처음이시네요!" 안내 팝업 표시

**SC-M6-007: 후킹페이지 유도** [High]
- Given: 안내 팝업 표시
- When: [후킹페이지 보러가기] 클릭
- Then: 랜딩페이지로 이동

**SC-M6-008: 바로 신청하기** [High]
- Given: 안내 팝업 표시
- When: [바로 신청하기] 클릭
- Then: 신청 플로우 진입

**SC-M6-009: 첫 모임 전날 환영 알림** [Critical]
- Given: 신규 회원의 첫 모임 전날
- When: 리마인드 Cron 실행
- Then: "첫 모임을 환영합니다!" 특별 알림 발송

**SC-M6-010: 기존 회원 전환** [Critical]
- Given: 신규 회원이 첫 정기모임 참여 완료
- When: 참여 완료 처리
- Then: is_new_member=false로 변경

**SC-M6-011: 첫 모임 후 다음 모임 안내** [High]
- Given: 신규 회원 첫 모임 종료
- When: 참여 완료 알림 발송
- Then: 후기 요청 + 다음 추천 모임 안내 포함

#### 사용자 가치
> "신규 회원이 자연스럽게 첫 모임에 참여하고 기존 회원으로 전환된다"

### Phase 3: 자격 체크 & 유도

**목표:** 정기모임 자격 조건을 체크하고 미충족 시 적절히 안내한다
**예상 소요:** 2~3일

#### 작업 목록

| # | 작업 | 산출물 | 완료 기준 |
|---|------|--------|----------|
| 3.1 | 자격 조건 정의 | 비즈니스 로직 | 신규 회원은 정기모임 불가 등 |
| 3.2 | 자격 체크 함수 | `lib/qualification.ts` | 조건 판별 |
| 3.3 | 자격 미충족 시 안내 UI | 모임 상세 | 안내 메시지 |
| 3.4 | 정기모임 목록 유도 | 안내 UI | "정기모임 먼저 참여해보세요" |
| 3.5 | 자격 충족 시 신청 허용 | 비즈니스 로직 | 정상 신청 가능 |
| 3.6 | 자격 상태 마이페이지 표시 | 마이페이지 | 현재 자격 안내 |

#### Scenario (검증 기준)

**SC-M6-012: 정기모임 자격 미충족** [Critical]
- Given: 신규 회원(정기모임 참여 이력 없음)
- When: 정기모임 신청 시도
- Then: "정기모임에 먼저 참여해보세요" 안내, 신청 불가

**SC-M6-013: 정기모임 자격 충족** [Critical]
- Given: 정기모임 참여 이력 있는 회원
- When: 정기모임 신청 시도
- Then: 정상적으로 신청 가능

**SC-M6-014: 자격 상태 표시** [High]
- Given: 마이페이지 접근
- When: 자격 섹션 확인
- Then: 현재 자격 상태 (신규/정규 회원) 표시

**SC-M6-015: 정기모임 유도** [High]
- Given: 자격 미충족 안내 표시
- When: "정기모임 보러가기" 클릭
- Then: 정기모임 목록 페이지로 이동

#### 사용자 가치
> "사용자가 자신의 자격 상태를 이해하고 적절한 모임에 참여할 수 있다"

### Phase 4: 안정화 & 출시

**목표:** 버그를 수정하고 성능을 최적화하여 정식 출시한다
**예상 소요:** 3~4일

#### 작업 목록

| # | 작업 | 산출물 | 완료 기준 |
|---|------|--------|----------|
| 4.1 | 버그 수정 (Beta 피드백) | 코드 수정 | 주요 버그 0건 |
| 4.2 | 성능 최적화 | 코드 개선 | Core Web Vitals 통과 |
| 4.3 | 이미지 최적화 | next/image | 로딩 속도 개선 |
| 4.4 | API 응답 시간 최적화 | 쿼리 개선 | 2초 이내 응답 |
| 4.5 | 에러 모니터링 설정 | Sentry 등 | 에러 추적 가능 |
| 4.6 | 에러 경계 (Error Boundary) | React | 크래시 방지 |
| 4.7 | 사용자 피드백 반영 | UI/UX 개선 | 주요 피드백 처리 |
| 4.8 | 최종 QA 테스트 | 테스트 결과 | 전체 기능 정상 |
| 4.9 | 프로덕션 환경 점검 | 체크리스트 | 환경 변수, DNS 등 |
| 4.10 | 출시 공지 준비 | 공지 내용 | 출시 안내문 |

#### Scenario (검증 기준)

**SC-M6-016: 주요 버그 없음** [Critical]
- Given: Beta 테스트 기간 수집된 버그
- When: 모든 Critical/High 버그 수정
- Then: 재현 불가, 정상 동작

**SC-M6-017: 성능 기준 충족** [Critical]
- Given: 프로덕션 환경
- When: Lighthouse 측정
- Then: Performance 70+, LCP 2.5초 이내

**SC-M6-018: 에러 모니터링** [High]
- Given: 에러 발생
- When: 사용자가 에러 경험
- Then: Sentry에 에러 기록, 알림 수신

**SC-M6-019: 전체 플로우 테스트** [Critical]
- Given: 신규 사용자 시나리오
- When: 가입 -> 모임 조회 -> 신청 -> 결제 -> 참여 완료
- Then: 모든 단계 정상 동작

**SC-M6-020: 프로덕션 배포** [Critical]
- Given: 모든 테스트 통과
- When: main 브랜치 배포
- Then: 프로덕션 환경에서 정상 동작

#### 사용자 가치
> "사용자가 안정적인 서비스를 경험하며 정식 출시가 완료된다"
